二、滑动窗口限流

    核心目的是防止突发流量（比如秒杀、高频接口调用）导致系统过载，整体是基于redis、AOP和自定义注解来实现的，解耦性强、配置灵活，而且能解决高并发下的限流准确性问题。

1.注解（标记限流接口）

自定义一个限流注解（比如 @RateLimit），作用是标记需要限流的接口，可以通过注解参数，灵活配置限流规则。

注解核心参数：

  • windowSize：滑动窗口大小（比如60秒）

  • limitCount：窗口内最大允许请求数（比如100次）

  • limitType：限流类型（用户限流还是ip限流）

2. AOP（拦截接口，执行限流逻辑）

利用 Spring AOP 的环绕通知，拦截所有添加了 @RateLimit 注解的接口，在接口执行前，先执行限流校验，校验通过才允许接口执行；校验失败，直接返回”请求过于频繁“。

3. Redis（实现滑动窗口计数，保证原子性）

这是限流的核心，用 redis 的 ZSet 实现滑动窗口，解决固定窗口限流的临界问题（比如固定60秒窗口，59秒和1秒各请求100次，实际2秒内200次，会突破限流），同时保证高并发下的原子性。

redis滑动窗口核心逻辑：

  • 生成唯一key：区分不同用户/IP的限流。

  • 添加当前请求时间戳：往 ZSet 中添加一条数据，value为请求唯一标识（比如 UUID），score 为当前时间戳（毫秒级）。

  • 清理过期请求：删除 zset 中，score 小于 “当前时间 - 滑动窗口大小” 的数据。

  • 统计当前窗口请求数：获取 zset 的长度（即当前窗口内的请求次数）。

  • 判断是否限流：如果请求数 > 注解配置的 limitCount，返回限流失败；否则，允许接口执行。

4.还了解什么限流算法呢

令牌桶算法（和滑动窗口限流比，允许合理的流量突发）

  • 系统维护一个固定容量的令牌桶

  • 按照固定速率匀速往桶里投放令牌

  • 桶满之后，多余令牌直接丢弃，不再存入

  • 客户端每来一次请求，必须先从桶里拿走 1 个令牌

  • 拿到令牌：正常放行执行业务

  • 拿不到令牌：直接拒绝 / 排队，实现限流

三、用户登录校验

session做法：

    Session 空间是服务器给每个用户单独开辟的一块内存空间。用户第一次访问服务器，会创建一个新的 Session，服务器返回给前端 SessionID，将其存在 Cookie 里。下次用户访问 → 带着 SessionID → 服务器找到属于他的那块 Session 空间。但是原生 Session 存储在单台服务器内存中，无法直接实现集群共享，多节点部署会出现登录状态丢失的问题。

redis做法：

登录链路 ：

  • 用户提交账号密码登录
  • 后端校验账号密码无误
  • 生成唯一会话标识 Token，组装用户登录信息
  • 将登录数据存入 redis，设置过期时间（Hash存储）
  • 把 Token 返回前端，前端存入 Cookie / 本地存储
  • 后续所有请求头部携带 Token
  • 后端第一层拦截器拦截所有请求，在前置拦截中获取 Token，查询 Redis 是否存在登录信息
  • 把查到的用户信息存入 ThreadLocal 本地线程变量，并刷新 Token 有效期
  • 在第二层拦截器中检查 ThreadLocal 中是否有用户信息，没有就拦截
  • 请求结束，清空 ThreadLocal，防止内存泄露、线程复用串数据
  • 登出直接删除 Redis 中对应会话数据

为什么用双层拦截器

    请求可以分为两大类，公共的接口和需要登录状态的接口，拦截器拦截需要登录状态的接口，如果设置一层拦截器，在登陆后，用户在长时间内访问公共接口，导致 Token 过期，需要重新登录，用户体验不好。

    因此设置两层拦截器，第一层为全局会话拦截器，拦截系统全部接口，无论公共接口还是权限接口，只要请求头携带有效 Token，就自动从 Redis 读取登录信息存入 ThreadLocal，同时刷新 Token 过期时长，保证用户持续在线操作时会话一直有效。第二层为登录权限拦截器，只针对性拦截需要登录授权的业务接口，仅校验 ThreadLocal 中是否存在用户信息，有则放行执行业务，无则直接拦截并提示未登录，公共接口直接跳过该层校验，无需登录也可正常访问。

JWT做法

  • 用户提交账号密码

  • 后端校验账号密码

  • 生成 JWT Token

    把 userId、用户名、权限、过期时间 直接放进 Token

    用 密钥签名，防止篡改

  • 直接返回 JWT 给前端

  • 前端存 Cookie /localStorage

  • 后续请求头携带 JWT

  • 拦截器获取 JWT → 校验签名、过期时间

  • 从 Token 里直接解析出用户信息

  • 放入 ThreadLocal

  • 放行业务接口
