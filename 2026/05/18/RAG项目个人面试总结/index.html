权限控制

这个项目的权限体系可以分成两条线：认证和授权。

一、认证：你是谁

    认证通过一个自定义的 JwtAuthenticationFilter 实现。这个过滤器做的事情是：

  • 从请求头中提取 JWT

  • 验证分两步：先查 Redis 确认这个 token 没被拉黑、没被登出，再校验 JWT 本身签名和过期时间

  • 刷新 Token，即签发一个新的 Token 给前端

  • 验证通过后，从 JWT 里解析用户信息（用户名和权限列表），设置到 SecurityContext 里

二、授权第一层：URL 路由控制

    静态资源、登录注册、WebSocket 端点 → permitAll() 直接放行。

  • 文件上传、文档下载、搜索、对话历史 → hasAnyRole(“USER”, “ADMIN”)，登录用户都能用
  • /api/v1/admin/** → hasRole(“ADMIN”)，只对管理员开放
  • 其余所有请求 → authenticated()，必须登录

    这是由 Spring Security 内置的 AuthorizationFilter 来执行的，它在我们的自定义过滤器之后运行，根据 SecurityContext 里有没有对应角色来决定是否放行

三、授权第二层：数据级权限控制

第一条分支：

    不需要资源鉴权的请求。 就是那些 URL 里没有明确资源 ID 的操作——比如上传分片、合并文件、获取文档列表、混合检索、查看个人信息等等。这些请求不需要验证”你有没有权限访问某个特定资源”，直接放行就行。但对上传和搜索这类接口，Controller 后续还需要知道是谁在操作，所以过滤器会顺手从 JWT 里把 userId 和 role 提出来，设到 request 属性上，Controller 通过 @RequestAttribute 直接拿。后续在 Service 层做数据过滤——比如搜索的时候，在 ES 的 filter 条件里拼上 userId=自己 OR isPublic=true OR orgTag IN 我的标签，从数据源头上就把没权限的结果过滤掉。

第二条分支：

    需要资源鉴权的请求。 也就是 URL 里带了资源 ID 的——比如查看某个文档、操作某个文件。这时候得从 URL 里把资源 ID 提取出来，去 FileUpload 表查出这个资源的 owner、orgTag 和是否公开，然后按优先级逐级判定。这套优先级是这样的：

  • 资源是公开的 → 直接放行，所有人可见

  • 资源没有 orgTag 或者属于 DEFAULT → 放行，相当于默认共享

  • 当前用户是资源拥有者 → 放行，自己的数据当然能看

  • 当前用户是 ADMIN → 放行，管理员拥有全部数据访问权

  • 资源 orgTag 以 PRIVATE_ 开头 → 拒绝。因为前面已经排除了拥有者和管理员，走到这里说明你跟这个私人资源没有关系

  • 用户的 orgTags 集合包含资源的 orgTag → 放行，同组织成员可以互相访问

  • 以上都不满足 → 403 Forbidden

检索增强生成

为什么要用混合检索呢？

单纯用向量检索的问题是：向量捕捉的是语义相似度，但对精确关键词不敏感。比如用户搜”2025年Q3财报”，向量可能召回了”2024年Q2财务分析”，虽然语义相关但年份完全错了。反过来，纯用 BM25 的关键词匹配又捕捉不到同义表达，比如用户问”怎么提升团队效率”，文档里写的是”优化组织协作方式”，BM25 一个词都对不上。所以这套系统把两者融合在一起，取长补短。

检索流程

  • 用户查询向量化。 把用户的自然语言问题发给 DashScope 的 text-embedding-v4 模型，转成一个 2048 维的浮点数向量

  • KNN 向量召回。 在 ES 的 knowledge_base 索引上用这个查询向量做 KNN 搜索。这里有一个关键设计——召回窗口不是 topK，而是 topK * 30。比如最终要返回 top-5 的结果，KNN 阶段会先召回 150 个候选。原因是向量相似度只是一个粗略的信号，后面还要经过 BM25 重排和关键词过滤，必须保证候选池足够大，才不会漏掉好结果。相似度计算用的是余弦相似度，因为余弦对向量的方向更敏感，不受绝对长度影响，适合语义比较

  • 关键词约束。 在 ES 的 bool.must 里加了一层 match 查询，要求候选文档 textContent 必须命中用户的查询词。这相当于一个硬约束——向量召回的候选里，内容跟查询完全无关的文本片段会被这一步过滤掉

  • 权限过滤。 在 bool.filter 层用三层 should（OR逻辑）限定数据范围：userId 是自己的、isPublic 为 true 的、orgTag 在自己组织标签集合里的。用户的有效标签包含层级
    继承——比如你在 TECH_DEPT 下，你的标签集合会包含 TECH_DEPT、它的父标签 COMPANY、以及 DEFAULT。这一步在 ES层面就完成了，不会把无权限的数据拉回应用层再过滤

  • BM25 Rescore 重排序——这是评分融合的核心。 ES 的 rescore 机制允许对 KNN 召回的结果用另一个评分公式重新算分。这里设置了两个权重：最终得分 = KNN 余弦相似度 × 0.2 + BM25 文本匹配分 × 1.0

Prompt 组装

可以处理excel表格吗？

局限性：Excel 的结构化信息会丢失——比如 A 列是”产品名”、B 列是”销售额”，Tika 提取出来只是一串文本，不会保留行列关系和数值关联。所以复杂表格里的数据查询精度不如纯文本文档

BM25原理

  • 词频 —— 查询词在这篇文档里出现了多少次：一个文档里出现”销售额” 5 次，肯定比出现 1 次的相关度高。但它做过饱和处理——出现 100 次跟出现 50 次分数差不多，不会让一个词无限抬高分数。

  • 逆文档频率 —— 这个词在整个文档库里有多稀有。查询里的”的”、”了”这种词，几乎每篇文档都有，IDF 极低，对评分贡献趋近于零。而”KNN向量检索”这种专有词，可能只有少数几篇文档包含，IDF 极高，一旦命中就是强信号。

  • 文档长度归一化—— 长文档天然比短文档命中更多词。不能因为长文档词多就永远排前面。BM25 会根据文档长度做惩罚，把长文档的优势拉平。
